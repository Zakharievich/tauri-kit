# Руководство по использованию ИИ в проекте tauri-kit

Этот документ определяет границы допустимых изменений, вносимых с помощью
ИИ-ассистентов (Cline и аналогичные), и обязательные проверки перед тем,
как считать задачу выполненной.

## Допустимые изменения через ИИ

- Рефакторинг существующего кода без изменения публичного поведения
  (переименования, извлечение функций, улучшение читаемости).
- Добавление новых UI-компонентов, хуков, сервисов в `src/`, не
  затрагивающих зоны риска (см. ниже).
- Написание и расширение тестов (`vitest`, `cargo test`, `pytest`).
- Обновление документации (`docs/`, `README_AGENT.md`) при изменении
  поведения соответствующих модулей.
- Исправление багов, линт-ошибок, типизации.
- Добавление некритичных фич UI (индикаторы состояния, стили, лейауты).
- Правки CI-конфигурации (`.github/workflows`), не меняющие набор
  обязательных проверок (lint/build/test) на более слабый.

## Зоны риска (требуют повышенного внимания и явного ревью)

Изменения в этих зонах допустимы через ИИ, но обязаны сопровождаться
явным описанием, что именно изменилось и почему, плюс прогоном полного
набора тестов:

1. **E2EE-ключ** (`src/services/e2eeService.ts`, `ExternalE2EEKeyProvider`).
   Ключ никогда не должен логироваться, кэшироваться или отправляться на
   сервер. Любое изменение здесь требует ручной проверки, что ключ
   передаётся только через `keyProvider.setKey()` и не покидает клиент
   иначе, чем out-of-band (docs/PLAN.md HIGH RISK 4.2).
2. **Генерация токенов** (`server/src/tokenService.ts`,
   `server/src/config.ts`). TTL строго ограничен (`TOKEN_TTL_SECONDS`,
   default 1 час), сервер stateless — токен никогда не хранится и не
   логируется на сервере. Важно: сервер **не** отслеживает повторное
   использование токена (одноразовость не реализована — намеренный
   компромисс stateless-дизайна, см. HIGH RISK 4.1); эндпоинт `/token`
   защищён только rate-limit'ом.
3. **Rust-команды** (`src-tauri/src/commands.rs`, `tauri.conf.json`
   capabilities/CSP). Любое расширение прав доступа к файловой системе или
   иным нативным ресурсам должно быть минимально необходимым и явно
   обосновано; недопустимо расширять capabilities "про запас" (HIGH RISK
   4.4).
4. **Конфиг агента** (`agent/.env.example`, `agent/main.py`,
   `agent/transcriber.py`, `agent/summarizer.py`, `agent/exporter.py`).
   Изменение флагов `ENABLE_STT`/`ENABLE_SUMMARY`, URL сервисов STT/LLM,
   контракта DataChannel-сообщений (`transcript_final`) должно сохранять
   graceful degradation и не ломать совместимость с
   `src/hooks/useTranscription.ts` (HIGH RISK 4.3).

Во всех этих зонах — никаких секретов/API-ключей в исходном коде, только в
`.env`-файлах, покрытых `.gitignore` (HIGH RISK 4.5).

## Обязательные проверки перед тем, как считать изменение готовым

```bash
pnpm build && pnpm lint
```

и все тесты:

```bash
pnpm test              # src/
cd server && pnpm test # server/
cd src-tauri && cargo test
cd agent && pytest     # если затронут агент
```

Изменение не считается завершённым, пока эти команды не проходят успешно.

## Правило: Plan Mode для изменений, затрагивающих > 3 файлов

Если предполагаемое изменение затрагивает **более 3 файлов**, ИИ обязан
сначала перейти в Plan Mode: описать план изменений, файлы, которые будут
затронуты, и получить подтверждение пользователя перед переходом к
реализации (Act Mode). Это предотвращает неконтролируемые каскадные правки
в зонах риска.

## Правило: обновление документации при смене модели STT/LLM

При изменении модели STT (`FASTER_WHISPER_MODEL`, по умолчанию `tiny`) или
модели саммари (`OLLAMA_MODEL`, по умолчанию `qwen2.5:0.5b`), а также при
смене самих движков (например, замена faster-whisper-server или Ollama на
другой сервис), обязательно обновить:

- [agent/README_AGENT.md](../agent/README_AGENT.md) — таблицу требований к
  серверу, переменные окружения, примечания по точности/производительности;
- [ARCHITECTURE.md](./ARCHITECTURE.md) — раздел про то, как агент
  подключается к комнате и передаёт транскрипт, если изменился сам
  механизм (не только название модели).

Несоблюдение этого правила считается неполным изменением и должно быть
отклонено при ревью.
